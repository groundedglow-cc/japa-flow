# JapaFlow 0.1.3 产品功能说明书（单词学习闭环版）

## 一、版本定位

v0.1.3 在 v0.1.2 的互动输入层基础上，优先补齐单词模块的学习闭环。

目标链路：

```text
学习行为 → 反馈 → 诊断 → 复习 → 进阶
```

本版本只聚焦单词闭环。语法、课文、练习后续沿用同一诊断模型扩展。

---

## 二、核心问题

当前单词模块已经有：

- 标准音播放
- 跟读自评
- 中文 recall
- 熟悉 / 不熟状态

但还缺少：

- 客观发音评价
- 多维能力诊断
- 明确掌握规则
- 针对弱项的复习队列

其中“跟读自评”反馈价值较弱，本版本改为远端模型发音评价。

---

## 三、单词状态模型

单词状态分为两层。

### 1. 主状态

主状态只能有一个：

```text
未学 / 学习中 / 待复习 / 已掌握
```

### 2. 诊断标签

诊断标签可以多个：

```text
听音弱 / 日文识别弱 / 发音不标准 / 流畅度不足
```

示例：

```json
{
  "mainStatus": "review",
  "diagnosticTags": ["听音弱", "发音不标准"]
}
```

---

## 四、掌握规则

一个单词进入“已掌握”，必须同时满足三项：

1. 完成发音预热，且 Azure Pronunciation Assessment 判定发音通过
2. 听音选日文正确
3. 日文选中文正确

任一能力未通过：

- 主状态为“待复习”
- 写入对应诊断标签

---

## 五、发音评价

### 服务

使用 Azure Speech Pronunciation Assessment。

前端只负责录音和上传，不保存 key。

后端读取环境变量：

```bash
AZURE_SPEECH_KEY
AZURE_SPEECH_REGION
```

注意：

- Azure Portal 显示的 `East Asia` 在 SDK/API 中应配置为 `eastasia`
- 用户给出的值需要确认是否是 Speech 资源的 Key，而不是资源 ID 或订阅 ID

### 第一版反馈粒度

只做：

- 通过 / 不通过
- 发音总分
- accuracy
- fluency
- completeness
- 原因标签

不做假名级可视化纠音。

---

## 六、单词学习流程

单词学习拆成两个阶段，避免用户刚看完单词后立刻测试，导致短期记忆污染测试结果。

### 阶段 1：全量发音预热

```text
展示单词
↓
播放标准音
↓
用户录音
↓
Azure 发音评价
↓
进入下一个词
```

所有单词都完成发音预热后，才进入系统测试。

### 阶段 2：全词表打乱测试

```text
听音选日文
↓
日文选中文
↓
随机打乱出现
↓
生成诊断
↓
进入下一词或复习队列
```

测试阶段不展示假名、完整中文解释、右侧词表或任何会泄露答案的状态信息。

### 干扰项策略

选择题优先使用预置相似干扰项，再回退到本课相似词。

干扰项来源包括：

- 字形相似
- 发音相似
- 语义相近
- 词类相同

---

## 七、复习机制

复习队列只包含非“已掌握”单词。

复习题型根据诊断标签决定：

- 听音弱 → 听音选日文
- 日文识别弱 → 日文选中文
- 发音不标准 / 流畅度不足 → 听标准音 → 录音 → 远端评价

用户可以进入语法和课文，但系统持续提示未掌握单词数量。

---

## 八、接口设计

### 发音评价

```text
POST /api/pronunciation/evaluate
```

请求：

- `wordId`
- `referenceText`
- `kana`
- `audio`：WAV 文件

响应：

```json
{
  "passed": true,
  "pronunciationScore": 86,
  "accuracyScore": 90,
  "fluencyScore": 82,
  "completenessScore": 100,
  "reasons": [],
  "raw": {}
}
```

---

## 九、验收标准

- 用户可以录制单词发音并提交评价
- Azure 返回结果后页面展示通过 / 不通过和原因
- 发音未通过时，单词不能进入“已掌握”
- 两类选择题任一失败时，单词不能进入“已掌握”
- 测试阶段不展示假名、中文解释、右侧词表等额外提示
- 选择题选项优先来自预置相似干扰项
- 诊断标签能正确展示在单词页和结果页
- 复习队列能根据诊断标签生成题型
- 旧版 localStorage 数据能兼容迁移

## 十、全局播放速度控制

页面顶部提供统一的播放速度切换入口，作用于当前页面所有音频行为，包括：

- 单词标准音播放
- 课文句子播放
- 语法例句播放
- 录音失败后的文本转语音回退

### 取值

```text
0.6x / 0.8x / 1.0x / 1.2x / 1.5x
```

### 交互要求

1. 播放速度控制必须固定在页面顶部的全局导航区域，用户在任一模块都能看到。
2. 用户切换速度后，当前正在播放的音频应立即按新速度继续播放，不需要等到下一次播放。
3. 切换后的速度需要持久化，刷新页面后仍保持上一次选择。
4. 语音合成播放和本地音频播放都必须统一受该控制影响。

### 验收标准

- 用户可以在顶部切换速度档位
- 当前播放音频切换速度后立即生效
- 刷新页面后仍保留上次速度选择
- 单词、课文、语法里的播放入口都遵循同一速度设置

## 十一、语法点侧边进度

语法模块左侧每一个语法点都应显示学习状态，让用户能直接知道当前语法点处于哪一步，而不是只能点进去后才判断。

### 状态

语法点至少区分以下状态：

- 未学
- 学习中
- 未通过
- 已学完

### 展示要求

1. 左侧每个语法点条目必须显示状态提示。
2. 每个语法点条目应有可感知的进度信息，例如 `x/y`。
3. 已完成、进行中、未通过等状态应在视觉上明显区分。
4. 用户无需进入详情页，就能从侧边列表判断哪些语法点已经学完，哪些仍需继续练习。

### 验收标准

- 左侧语法点可直接识别完成状态
- 未学、未通过、学习中、已学完四类状态在列表中可区分
- 进度数值可反映当前语法点的学习推进情况
- 状态变化后列表应即时刷新
