# JapaFlow PRD v0.1.10 - 课内 AI 学习助手

## 1. 背景

JapaFlow 当前以“按课次”组织学习：单词预热、语法、课文、练习、错题集、结果、音频。学习者在阅读课文或做语法练习时，常常会冒出一些“一句话能解释完”的疑问：

- 这句话里这个助词为什么用 `に` 不用 `で`？
- 这个语法点和之前学过的 `てくれる` 有什么区别？
- 这个汉字这里读作什么、为什么不读训读？
- 这道练习题为什么我做错了？正确思路是什么？

目前用户只能跳出 app 去查工具书或搜索引擎，体验断裂、上下文丢失。`ai-assistant-qa.md` 已经从产品形态层面给出了“悬浮入口 + 选句弹窗 + 侧边对话面板”的推荐方案。本 PRD 的目标是把该方案落到 JapaFlow 当前的工程结构和学习场景里，给出可执行的最小版本（MVP）和后续扩展路径。

## 2. 目标

v0.1.10 目标是引入一个**贴合课文/语法/练习场景的 AI 学习助手**，做到三层递进体验：

1. **一看即懂**：选中课文中的句子或词，可在原地呼出轻量解释卡片。
2. **一追问就深**：在卡片上一键转入侧边对话面板进行多轮追问。
3. **不问时静默**：助手以悬浮入口形式存在，不打扰主学习流。

并且：

- 助手默认带上当前课次上下文（课号、当前所在 tab、当前句子或练习），无需用户重复粘贴。
- 所有 AI 调用走自建 dev server（`server.mjs`）转发，不在前端裸暴露密钥。
- 与现有发音评测、音频生成等能力相互独立，不阻塞主链路。

## 3. 非目标

- 不在本版本接入语音对话（说出去 → AI 回答），只做文本对话。
- 不为助手单独建路由页或入口卡片，助手是“贴在已有页面上的浮层”。
- 不实现自动主动弹窗、自动打断阅读的形式。
- 不替换或下线现有发音评测、音频生成、错题集等能力。
- 不在 v0.1.10 把助手能力接入“课程初始化页”（init 流程目前已有自己的 AI 流水线）。
- 不持久化整段对话到课程数据 JSON；对话历史仅留在本地 storage。
- 不在本版本支持多设备会话同步。

## 4. 适用场景

助手在以下页面默认可用：

- `/lesson/:id/text`（课文）
- `/lesson/:id/grammar`（语法）
- `/lesson/:id/exercises`（练习）
- `/lesson/:id/wrongbook`（错题集）
- `/lesson/:id/vocab`（单词预热）

在首页 `/`、`/lesson/:id`（导航汇总页）和音频/结果页**不显示**助手入口，避免无意义的浮层。

## 5. 用户流程

### 5.1 悬浮入口

- 在适用页面的右下角常驻一个气泡图标（移动端略上移以避开 home indicator）。
- 点击图标，从右侧滑出 **侧边对话面板**：
  - 桌面端：宽 360–420px 的右抽屉，主区不被遮挡。
  - 移动端：从下方上推 70% 视高的半屏抽屉。
- 面板顶部展示当前课次上下文徽章，例如：`第 29 课 · 课文 · 第 4 句`。
- 输入框默认聚焦，支持自由提问。

### 5.2 选句解释

- 用户在课文区或语法例句区拖选/长按选中一段文本（一个词到一整句皆可）。
- 选中结束后，在选区附近浮出一个小工具条，含两枚按钮：
  - `🔍 解释`：调用 AI 生成轻量解释卡片。
  - `📖 详细分析`：直接把该句带入侧边面板深度追问。
- 仅当选区落在课文文本节点内才显示工具条；落在已有按钮/控件上不触发。

### 5.3 解释卡片

- 卡片以 popover 形式贴在选区附近，自动避让，不遮挡选中文本。
- 卡片内容分级展示（流式渲染）：
  1. **简明翻译**：中文意思。
  2. **关键点标签**：例如“被动语态 / 助词 `に` / 拟态词”，至多 3 个，点击展开一两句说明。
  3. 底部操作栏：`📖 详细分析` `➕ 加入生词本（仅在选区是单个未收录词时显示）` `📋 复制`。
- 关闭条件：点击页面其他区域、按 Esc、滚动主区、再次点击🔍。

### 5.4 卡片 → 对话面板

- 点击 `📖 详细分析` 后：
  1. 若侧边面板未打开，则自动滑出。
  2. 面板输入框自动注入：`请详细解释「<原句>」` + 当前课次/句号上下文。
  3. 自动提交一次，生成多轮可追问的结构化回答。
  4. 对话被记入本地历史。

### 5.5 错题追问

- 在错题集页，每条错题卡片末尾增加 `🤖 问一下` 按钮。
- 点击后打开侧边面板，自动注入题面、正确答案、用户当时的答案，请求 AI 解释“为什么我错了”。

## 6. 信息架构与上下文

助手每次调用 AI 时会附带一个**结构化的上下文对象**，由前端组装，server 转发时不做修改：

```json
{
  "appVersion": "0.1.10",
  "lessonId": 29,
  "lessonTitle": "第29课",
  "page": "text",
  "tab": "basic",
  "selection": {
    "text": "森さんは李さんに北京を案内してもらいました。",
    "sentenceId": "s3"
  },
  "wrongAnswer": null,
  "userQuestion": "这里为什么用 もらいました 不用 くれました？"
}
```

字段说明：

- `selection.text`：用户选中或当前句的原文，最多 500 字符。
- `selection.sentenceId`：若能从 DOM 反查到 lesson 数据里的句号，则带上，便于后续模型基于句子的语法标签做更准确回答。
- `wrongAnswer`：仅在错题集追问场景下出现，结构与 `data/lessons/lessonXX.json` 中练习答案字段对齐。
- `userQuestion`：用户在面板输入框中的提问。

> 不发送整本 lesson JSON。模型上下文窗口和延迟都不允许这样做。仅在用户主动追问“这个语法点和之前哪些课讲的有什么区别”时，再追加当前课的语法点摘要（最多 30 行）。

## 7. 前端实现

### 7.1 模块拆分

考虑现有 `app.js` 已经 6700+ 行，新增内容**不直接堆进 `app.js`**，按以下方式拆分：

- `app/ai-assistant/floating-entry.js`：悬浮气泡入口和挂载点。
- `app/ai-assistant/selection-popover.js`：选区监听、工具条、解释卡片。
- `app/ai-assistant/chat-panel.js`：侧边对话面板和会话状态。
- `app/ai-assistant/context.js`：从当前路由和 DOM 组装上下文对象。
- `app/ai-assistant/api.js`：调用 `/api/ai/*` 的封装，统一处理流式响应、错误、取消。
- `styles/ai-assistant.css`：样式独立，使用 `ja-` 前缀避免冲突。

`app.js` 只在 `render()` 结尾调用一次 `mountAIAssistant(route())`，由助手模块自行判断当前路由是否应显示入口。

### 7.2 状态管理

- 对话历史按 `lessonId` 分桶，键为 `japaflow.aiChat.lesson<ID>`，存 localStorage。
- 单条消息结构：`{ id, role: "user" | "assistant", content, createdAt, context }`。
- 每个 lesson 仅保留最近 20 条消息；超出滚动丢弃，避免 storage 膨胀。
- 用户可在面板顶部菜单点击“清空当前课对话”。

### 7.3 流式渲染

- AI 接口必须支持 SSE 流式输出。
- 卡片和面板都在收到首个 chunk 后立即展示骨架文字，逐字追加。
- 切换页面、关闭面板、再次提问会取消当前请求（`AbortController`）。

### 7.4 与发音/音频能力的边界

- 助手不调用 TTS。如果未来需要朗读 AI 回复，单独迭代。
- 用户当前播放音频时，打开助手面板不打断音频；助手也不主动暂停音频。

## 8. 后端实现

### 8.1 新增接口

在 `server.mjs` 中新增三个端点：

| 路径 | 方法 | 说明 |
| --- | --- | --- |
| `/api/ai/explain` | POST | 选句轻量解释，SSE 流。Body: `{ context }` |
| `/api/ai/chat` | POST | 多轮对话，SSE 流。Body: `{ context, history, userQuestion }` |
| `/api/ai/health` | GET | 探活，返回当前 provider 名和模型 ID |

### 8.2 配置项

复用现有 env 注入风格：

```
AI_PROVIDER=openai            # 可选 openai / azure / qwen，默认 openai
AI_API_KEY=...
AI_MODEL=gpt-4o-mini
AI_BASE_URL=https://api.openai.com/v1   # 可选，便于走代理
AI_REQUEST_TIMEOUT_MS=30000
AI_MAX_TOKENS=600              # explain 用；chat 默认 1200
```

- 缺少 `AI_API_KEY` 时，`/api/ai/health` 返回 503，前端入口直接隐藏并在控制台打一行 warning。
- 不在 iOS 打包产物中走该接口；Capacitor App 端的助手在 v0.1.10 仅支持 dev server 模式（详见 §11 风险）。

### 8.3 Prompt 模板

- **system prompt（explain）**：约束模型只输出 JSON 结构 `{ translation, tags: [{label, detail}], note? }`，便于卡片渲染。
- **system prompt（chat）**：明确身份是“日语学习助手”，要求按学习者水平（用户在上下文中标注的课程级别）调整难度，回复中文，必要时给假名。
- 所有 prompt 模板单独放在 `server/ai-prompts.mjs`，便于和模型一起调参。

## 9. 性能与成本

- 解释卡片采用小模型（如 `gpt-4o-mini`/同档位），目标首字 < 800ms、整段 < 3s。
- 对话面板可用同档模型；如未来需要更强推理，做按需切换。
- 单次请求 token 上限默认 600（explain）/ 1200（chat），上下文 + 历史合计 ≤ 2K tokens。
- 服务端做简单 in-memory 速率限制：同一 IP 每分钟最多 30 次请求，命中限流返回 429 + 友好提示。

## 10. 隐私与合规

- 助手只发送：当前选区文本、当前课号、用户输入。**不发送**用户的发音录音、设备信息、错题历史中的个人答案以外的隐私数据。
- 在面板首次打开时展示一次性提示：“你的提问将发送至第三方大模型（OpenAI 等）以生成回答，请勿输入隐私信息。” 用户点击“知道了”后不再展示。
- 默认不上报对话内容到任何分析平台。

## 11. 风险

### 11.1 技术风险

- `app.js` 已 6700+ 行，再注入助手代码若不拆模块会进一步劣化可维护性。需先落地 §7.1 的拆分。
- iOS Capacitor 打包后无 Node server，助手能力暂不可用，需要在入口处做能力探测，避免出现“点了没反应”的情况。
- 选区 popover 在移动端 WebView 与原生选择菜单的冲突需要专门处理（参考 ai-assistant-qa.md §4 的 Selection API 思路）。

### 11.2 产品风险

- AI 解释错误会反过来误导学习者，尤其是助词、敬语、授受动词这种本课重点。需要在解释卡片角标提示“AI 解释，仅供参考”。
- 选句弹窗的“两步操作”可能让习惯划词翻译的用户觉得多余。需要在设置中提供“选中后直接展示解释”的高效模式开关（默认关闭）。

### 11.3 成本风险

- 高频选句调用会拉高 AI 调用成本。MVP 阶段需要一个最小的本地节流：同一选区 30 秒内不重复请求，命中缓存。
- 若 dev server 被局域网内多人共享，需要在 README 提醒打开 `AI_RATE_LIMIT` 配置。

## 12. MVP 范围

第一阶段只交付以下能力：

1. 课文页右下角悬浮气泡，点击展开侧边对话面板（桌面端右抽屉、移动端半屏）。
2. 课文页选中文本后浮出工具条，含 `🔍 解释` 和 `📖 详细分析` 两个按钮。
3. 解释卡片支持流式渲染，并提供“详细分析”跳转到对话面板。
4. 对话面板支持文本提问、流式回答、按课分桶的本地历史、清空当前课对话。
5. server.mjs 新增 `/api/ai/explain`、`/api/ai/chat`、`/api/ai/health` 三个端点，支持单一 provider（默认 OpenAI 兼容）。
6. 缺少 API key 时入口完全隐藏，不影响其它学习流程。

不在 MVP：

- 语法页和练习页的选句、错题集追问按钮。
- 高效模式、加入生词本、复制按钮等细节。
- 多 provider 切换、客户端缓存层、对话搜索。

## 13. 验收标准

- 在课文页选中任意一句话，能在 1 秒内浮出工具条，点击解释后 3 秒内出现完整卡片。
- 卡片关闭、面板收起、切换 tab 时正在进行的请求会被取消，不出现“关闭后内容仍在追加”的现象。
- 对话历史在刷新页面后仍能恢复，仅限当前课次的历史。
- 未配置 `AI_API_KEY` 时，所有 AI 入口在前端完全不渲染，控制台仅一行 warning，不影响其它页面功能。
- 在 iOS Capacitor 打包产物中打开 app，不会出现 AI 入口或与之相关的报错。
- `/api/ai/health` 返回当前 provider、模型 ID 和 `ok: true|false`。

## 14. 推荐实施顺序

1. 在 `app/` 下新增 `ai-assistant/` 目录骨架，在 `app.js` 末尾接入 `mountAIAssistant(route())`，先渲染一个空白悬浮气泡。
2. 在 `server.mjs` 加 `/api/ai/health`，验证 env 注入和前端能力探测链路通畅。
3. 实现 `/api/ai/chat` + 侧边面板，先把面板内文本对话跑通（不带选区联动）。
4. 实现选区监听 → 工具条 → `/api/ai/explain` → 解释卡片。
5. 打通“卡片详细分析”→ 自动注入侧边面板并提交。
6. 加上本地历史持久化、按课清空、节流、错误兜底。
7. 把 §12 列出的非 MVP 项排进 v0.1.11 候选清单。
